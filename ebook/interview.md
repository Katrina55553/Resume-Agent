## Q1: LangGraph 状态机的节点和边怎么设计？"回答质量"怎么量化？"三层防死循环"是哪三层？

**面试官原话**：节点和边是怎么设计的？条件边的路由逻辑是什么？"质量"怎么量化判定？三层防死循环具体是哪三层？LLM 在节点间来回跳转死循环怎么防？

### 我的回答

**先说节点和边。**

我的状态机有 4 个节点：`generate_question`（生成问题）、`collect_answer`（收集回答）、`evaluate_answer`（评估回答）、`generate_report`（生成报告）。

核心循环是：生成问题 → 收集回答 → 评估 → 条件分支。条件边只有一条，在 evaluate 之后，路由函数只返回两个目标——`generate_question`（继续问）或 `generate_report`（结束）。

这里有个设计点我想强调：**graph 层只管"继续 or 结束"，不管"是追问还是切换存疑点"**。follow_up 和 next_point 的区分在 evaluate 节点内部决定，通过 `decision` 字段透传给上层 WebSocket handler。这样图保持简单，加节点不影响已有边。

**再说质量怎么量化。**

用 LLM 从 5 个维度评估：真实性、深度、完整性、逻辑性、技术准确性，输出 0-100 的综合分。评估节点支持 Tool Calling——LLM 会主动调知识库检索工具验证候选人说的技术对不对。

决策阈值是 **75 分**：分数 ≥75 或追问到 3 轮上限，当前存疑点判定 resolved，切下一个点；否则继续追问。

但光看分数不够，加了两个硬性压制：LLM 判定回答不可信（`credible=false`）→ 分数压到 ≤45；回答太短（<50 字符）→ 分数压到 ≤40。确保"洋洋洒洒但没料"和"一句话敷衍"都蒙混不过去。

**最后说三层防死循环，这是 Agent 最容易翻车的地方。**

**第一层，节点级——单点追问上限。** 同一个存疑点最多追问 3 轮，到 3 轮强制切下一个点。防的是"LLM 死活不满意候选人回答，无限追问"。

**第二层，图级——会话硬熔断。** 路由函数检查四个全局上限：连续 LLM 异常 3 次、单场最多 20 个问题、Token 上限 5 万、单次 LLM 调用 15 秒超时。只要触上限就强制结束，不管 evaluate 返回什么。防的是"节点决策层失灵"。

**第三层，LLM 降级——避免 LLM 不确定性。** LLM 不可用、超时、JSON 解析失败时，自动降级到规则评分，按回答长度给分。规则逻辑确定，永远不会让状态机卡住。

**总结一句话**：节点级防单点死循环、图级防会话级失控、降级层防 LLM 不确定性——三层从局部到全局、从概率到确定，把 LLM 这种不可控的东西框在一个可预测的状态机里。这是我对 Agent 工程化最核心的理解。

---

### 可能的追问准备

- **"为什么阈值是 75 分？"** —— 经验值，实际跑下来 75 分能区分"有细节支撑"和"泛泛而谈"，可调。
- **"规则评分会不会误判？"** —— 会，长度不代表质量，但作为 LLM 挂掉时的兜底够用，总比流程卡死强。
- **"为什么 MAX_FOLLOW_UP 是 3？"** —— 面试场景下 3 轮足够判断候选人懂不懂，再问下去浪费双方时间。
- **"为什么不把 follow_up/next_point 做成条件边的两个目标？"** —— 图结构塞业务逻辑会变复杂，加节点要改边；放节点里图保持简单，业务逻辑也好测。

## Q2: LLM 评分的稳定性和可信度问题

**面试官追问**：你的评估器本身也是 LLM 调用。同一个回答跑两次分数可能差 10 分，75 分阈值在波动面前很脆弱。`credible=false` 本身也是 LLM 判的，用不稳定的判定修正不稳定的判定，是不是循环依赖？为什么没做 majority vote？

### 我的回答

您这个问题问到点子上了，我先承认——这是整个系统最薄弱的一环，LLM 评分不稳定是 Agent 系统的本质矛盾，我没有彻底解决，但做了四件事把它限制在可接受范围。

**第一，temperature 压到 0.3。** 评估节点是整个系统里温度最低的，生成问题用 0.7、闲聊用 0.9，评估刻意压低随机性。这是最直接的降波动手段。

**第二，引入客观锚点。** 评估节点支持 Tool Calling，LLM 会调知识库验证候选人说的技术对不对。比如候选人说"B+ 树适合范围查询"，LLM 调知识库查到这条是对的，这就不是主观判断了，是客观事实。工具结果回喂给 LLM，相当于外部约束，比纯主观评分稳。

**第三，用布尔判定兜连续值。** 我承认 `credible` 也是 LLM 判的，但它和 score 不是一回事——score 是 0-100 容易飘，credible 是布尔值，只有"明显不可信"才标 false，相当于否决权而非评分权。用稳的去约束飘的，逻辑成立。而且我还有 `answer_too_short` 这个 100% 规则化的判定，完全不依赖 LLM。

**第四，三层防死循环本身就是对评估不稳定的兜底。** 即使某次评分抽风切错了点，最坏后果是这个存疑点没问透，不影响整体面试完成。我接受这个代价——模拟面试不是司法判决，偶尔飘一次可接受。

**为什么没做 majority vote？** 成本和延迟。单场面试 10-15 分钟，每个回答跑 3-5 次评估，延迟翻倍 Token 翻倍，体验差。

**我承认的局限：** 如果做生产级，我会做两件事——一是离线跑评估一致性测试，拿 100 条回答跑 10 次看方差，方差大的加规则约束；二是关键决策（最终报告）用更强模型做 judge，单次评估用轻量模型控成本。

**总结**：LLM 评分不稳定我没法消灭，但用低温、客观锚点、布尔压制、三层兜底，把它限制在"偶尔切错一个点"的范围内，不让它击穿整个流程。

---

### 可能的追问准备

- **"score 阈值 75 是怎么定的？"** —— 经验值，跑下来 75 分能区分"有细节支撑"和"泛泛而谈"。其实 75 分不是死线，3 轮上限也是切点，两个条件是 OR 关系。
- **"Tool Calling 自己失败了怎么办？"** —— 工具执行 try-catch，失败返回错误字符串，LLM 看到错误会换方式回答，不影响主流程。
- **"为什么 temperature 0.3 不是 0？"** —— 0 太死板，评估需要一点灵活性判断"有创意的回答"；0.3 是 trade-off。

## Q3: RAG 三级检索的结果怎么融合？向量和全文各自擅长什么？

**面试官原话**：三级检索的融合策略是什么——RRF、加权分数、还是去重拼接？为什么选这个？向量检索和全文检索各自擅长什么？"消息队列怎么保证不丢"和"Kafka 的 ISR 机制"分别哪一路能命中？

### 我的回答

**融合策略用 RRF（Reciprocal Rank Fusion），但只融合向量+ES 两路，关键词作为兜底。**

具体逻辑：向量检索和 ES 全文**并行召回**，各自返回 top-2K，然后用 RRF 公式 `score = Σ 1/(k + rank)` 融合排序，取 top-K。k 取 60，论文经验值。

**为什么只用 RRF 不用加权分数？** 三路检索器的分数量纲不一致——向量是余弦相似度 0~1，ES 是 BM25 分数可能 0~20。加权融合要先归一化，归一化方法又是调参坑。RRF 只看排名不看绝对分数，量纲无关，省了归一化。

**为什么关键词不参与融合？** 关键词是子串匹配，几乎所有结果都能命中，"排名"权重虚高会污染融合分数。所以只让向量+ES 融合，关键词在前两路都挂了时才兜底。

**RRF 的核心价值**：同一条结果被两路都命中，分数累加排名更靠前——多路共识的结果更可靠。这是融合比单路召回强的本质。

**降级链也保留了**：我做了 `RETRIEVAL_MODE` 配置，支持 `fusion`（默认，双路 RRF）/ `cascade`（串行降级老逻辑）/ `vector_only` 三种模式，便于 A/B 测试和回滚。

---

**第二个问题，向量和全文各自擅长什么。**

**向量检索擅长语义相关但字面无关的查询。** 比如候选人简历写"我用过 Kafka"，面试官问"消息队列怎么保证消息不丢"——"消息队列"和"Kafka"、"不丢"和"ack 机制"字面差很远，全文检索抓不住，但向量把它们映射到相近的语义空间，能命中。本质是 embedding 把词映射到向量空间后，同义词、上下位词距离近。

**全文检索擅长术语精确匹配。** 比如"Kafka 的 ISR 机制"、"MVCC"、"B+ 树"——这些是专有名词，就是字面那几个字，向量检索反而可能因为语义相近漂到别的地方（比如 ISR 漂到"消息可靠性"的泛泛内容），全文检索字面精确命中更准。

**关键词匹配是兜底，前两个都挂了才用。** 零依赖、零延迟、永远不会挂。LLM 和 ES 都不可用时保证流程能走完。

**总结**：向量管语义、ES 管字面，双路 RRF 融合取共识；关键词管兜底，前两路都挂了才上。三种检索器各司其职，融合策略让多路共识的结果浮上来。

---

### 可能的追问准备

- **"为什么不用向量数据库？"** —— 32 条数据用内存列表 + 余弦相似度，零依赖零延迟。上向量数据库（ChromaDB、Milvus）要部署、要维护、要索引，收益不抵成本。数据量上千再换。
- **"RRF 的 k 为什么是 60？"** —— 论文经验值。k 越大排名靠后的结果分数差距越小（被压缩）；k 越小头部结果优势越明显。60 是平衡点，既不让第 1 名压倒一切，也不让排名 10 和排名 1 没区别。数据量小没做实测调优。
- **"embedding 模型为什么用 deepseek-embedding？"** —— 复用 LLM 厂商的 embedding 接口，不用额外部署模型。中文场景效果够用，且和 DeepSeek LLM 配套。
- **"向量检索的 top_k 为什么是 3？"** —— 注入 LLM prompt 的上下文不能太长，3 条已经够 LLM 参考。top_k 太大会稀释相关性、增加 token 成本。注意融合时每路召回 top_k * 2，给融合留余量。
- **"如果向量检索返回的是垃圾结果怎么办？"** —— RRF 融合能通过多路投票消除单路噪声——向量返回垃圾但 ES 命中好结果，融合后 ES 的好结果排名更靠前。这是融合比降级强的关键。
- **"为什么保留 cascade 模式？"** —— A/B 测试和回滚。融合是新版默认，但老逻辑不能丢，万一融合效果不如降级（虽然理论上不应该），一行配置切回去。生产环境必须有回滚手段。

## Q4: MCP 协议封装——核心机制、和 Function Calling 的区别、7 个工具怎么划分？

**面试官原话**：MCP 协议的核心机制是什么？和普通 Tool Calling（OpenAI Function Calling）有什么本质区别？为什么用 MCP 不直接暴露 REST API？7 个工具接口是哪 7 个？工具粒度按业务实体切还是按操作类型切？有没有粒度过粗或过细的问题？

### 我的回答

**先说 MCP 的核心机制和与 Function Calling 的区别。**

MCP（Model Context Protocol）是 Anthropic 提出的标准协议，本质是**把"AI 调用工具"这件事标准化**。它和 Function Calling 的核心区别是**解耦层次不同**：

- **Function Calling 是 LLM 能力**：LLM 自己决定调哪个函数、传什么参数，应用代码里写函数实现。函数定义和实现都绑死在应用里。
- **MCP 是协议层**：定义了"如何发现工具、如何调用、如何返回"的通信标准。工具实现独立成 Server，LLM 客户端通过协议发现并调用。

**最直观的区别**：用 Function Calling，我换个 LLM 厂商可能要改函数定义格式（OpenAI 和 DeepSeek 格式就不一样）；用 MCP，工具 Server 不动，换个支持 MCP 的客户端（Claude Desktop、Cursor、Cline）就能直接用。**MCP 让工具层和 LLM 层彻底解耦。**

**为什么用 MCP 不直接暴露 REST API？** 三个原因：

**第一，AI 直接调 REST API 要自己拼 URL、解析 HTTP 响应、处理错误码，不稳定。** MCP 用 JSON-RPC，输入输出都是结构化 JSON，AI 友好。

**第二，REST API 没有"工具发现"机制。** AI 怎么知道有哪些接口？得人在 prompt 里写死。MCP 有 `list_tools` 协议方法，客户端连上来自动发现所有工具和 schema。

**第三，REST API 要自己处理鉴权、限流、并发。** MCP Server 是独立进程，stdio 通信，天然隔离，安全性更好。

---

**再说 7 个工具怎么划分。**

我的 7 个工具是：`diagnose_resume`、`start_interview`、`answer_interview`、`skip_question`、`end_interview`、`get_report`、`list_sessions`。

| # | 工具名 | 业务阶段 | 必填参数 | 可选参数 | 干啥 |
|---|--------|---------|---------|---------|------|
| 1 | `diagnose_resume` | 上传+解析+诊断 | `file_path` | — | 传文件路径，内部跑完解析+诊断，返回结构化简历+存疑点 |
| 2 | `start_interview` | 开始面试 | `session_id` | `selected_point_ids` | 初始化面试状态，生成第一个问题；可选指定面试范围 |
| 3 | `answer_interview` | 提交回答 | `session_id`, `answer` | — | collect→evaluate→条件分支，返回下一个问题或最终报告 |
| 4 | `skip_question` | 跳过存疑点 | `session_id` | — | 当前点标 skipped，index+1，生成下一个问题 |
| 5 | `end_interview` | 提前结束 | `session_id` | — | 整场面试结束，基于已有评估生成报告 |
| 6 | `get_report` | 查看报告 | `session_id` | — | 读 `interview_states.report_json`，面试后随时可查 |
| 7 | `list_sessions` | 列出会话 | — | — | 唯一无参数工具，AI 用来发现有哪些历史会话 |

**几个关键设计点：**

- **`diagnose_resume` 合并了上传+解析+诊断三步**——早期拆三个工具 AI 经常调完 upload 不调 parse，流程卡住，合并后一个工具一气呵成。
- **`skip_question` 独立成工具**而不是 `answer_interview` 的特殊参数——"跳过"和"回答"是语义完全不同的用户动作，AI 决策更清晰。
- **`end_interview` 和 `get_report` 分开**——end 是动作（生成报告），get 是查询（读报告）。面试结束后用户随时能再查。
- **`answer_interview` 一个工具一次调用拿完整结果**——输入就 2 个参数，输出是下一个问题或最终报告，AI 不用连环调。

**划分原则是按业务流程切，不是按操作类型切。** 理由：

**按操作类型切（CRUD 思路）会过细。** 比如如果按"查询/创建/更新"切，我会切出 `create_session`、`get_session`、`update_session_status`、`create_interview_state`、`update_interview_state`... 一堆工具，AI 根本不知道什么场景调哪个。AI 需要的是"动作语义"，不是"数据库操作语义"。

**按业务流程切，每个工具对应一个用户动作。** 用户动作是：上传简历→诊断、开始面试、回答问题、跳过、结束、看报告、列会话。每个工具的 description 写清楚"什么场景调我"，AI 一看就知道。

**粒度把握的核心**：工具的输入参数要少而明确，输出要完整可消费。比如 `answer_interview` 输入就 session_id + answer 两个参数，输出是下一个问题或最终报告——AI 调一次能拿到完整结果，不用连环调。

**遇到的粒度问题**：早期我把 `diagnose_resume` 拆成了 `upload_resume` + `parse_resume` + `diagnose` 三个工具，AI 经常调完 upload 不调 parse，流程卡住。后来合并成一个 `diagnose_resume`，输入文件路径，内部一气呵成跑完三步。**粒度过细会让 AI 决策负担过重，合并非原子业务步骤是正确的。**

**总结**：MCP 解决的是工具层和 LLM 层的解耦，7 个工具按业务流程切，粒度对齐"用户动作"——这是 AI 友好的工具设计。

---

### 可能的追问准备

- **"MCP 和 LangChain Tools 比呢？"** —— LangChain Tools 是 SDK 级别的封装，工具和 LLM 调用代码在同一进程；MCP 是协议级别，工具是独立 Server 进程。MCP 更重但更解耦，LangChain 更轻但绑框架。我的项目同时用了：内部 LangGraph Agent 用 LangChain Tool Calling，对外暴露用 MCP。
- **"为什么不用 REST + OpenAPI schema 让 AI 调？"** —— OpenAPI schema 是给人看的文档，MCP 的 tool schema 是给 AI 看的结构化定义。OpenAPI 描述 HTTP 细节（method/path/status），AI 不需要关心这些；MCP 抽象掉了传输层，AI 只看 name/description/inputSchema。
- **"stdio 通信有什么局限？"** —— stdio 是本地进程通信，不能跨网络。MCP 也支持 SSE 传输做远程，但我用 stdio 是因为工具和 LLM 客户端在同一台机器，简单可靠。
- **"工具调用失败怎么处理？"** —— `call_tool` 里 try-catch，失败返回 `TextContent` 描述错误，AI 看到错误文本会换方式重试或换工具。不抛异常给 MCP 框架，保持协议层稳定。
- **"7 个工具够吗？以后扩展怎么办？"** —— 够用。扩展点在内部实现，不在工具数量。比如 `get_report` 以后支持导出 PDF，是内部逻辑变化，工具接口不变。工具数量越多 AI 决策越难，宁少不多。

## Q5: 你用 AI 开发的流程是怎样的？结合这个项目说说

**面试官原话**：你用 AI 开发的流程是怎样的？结合这个项目说说。

### 我的回答

**先说我对 AI 辅助开发的定位。**

我把 AI 当成一个"高年级的结对编程伙伴"——它能帮我写样板代码、查 API、做代码 review，但**架构决策和验收标准在我手里**。我不会让 AI 从零设计整套系统，但我会让它帮我实现我已经想清楚的模块。

**结合这个项目，我的流程分四步。**

**第一步：我自己想清楚架构和关键决策，写成文档。**

这个项目动手前我先把 AGENTS.md 写了——技术栈选型、核心架构、用户流程、面试 Agent 的节点循环图、WebSocket 通信协议、敏感信息脱敏的三层策略。这些是我自己定的，不是 AI 生成的。

为什么这步自己做？因为**架构决策的依据是业务约束和权衡**，AI 不知道我的部署环境（阿里云、要过代理）、不知道我的成本约束（DeepSeek 而不是 GPT-4）、不知道我对这个项目的学习目标（想深入理解 LangGraph）。这些只有我能判断。

**第二步：把模块拆细，让 AI 写实现。**

架构定了之后，每个模块的输入输出是清晰的。比如 `rag.py` 的 `retrieve_context` 函数——我告诉 AI："输入是 query 字符串，输出是检索到的知识条目列表，策略是向量检索优先、ES 兜底、关键词最后降级，参考 knowledge/ 下的 JSON 格式"。AI 写出来的代码我过一遍，不对的地方我自己改。

这一步 AI 的价值是**省掉样板代码的打字时间**。比如 Pydantic model、FastAPI 路由、Dockerfile 这些套路化的东西，AI 写得又快又对。但 LangGraph 的状态机转移逻辑、评估节点的 prompt 设计这种核心逻辑，AI 写的我不放心，要么自己写要么 AI 写完我逐行 review。

**第三步：用 AI 做 code review 和追问。**

写完一个模块，我会把代码贴给 AI 让它挑问题。比如 RAG 检索我写完三级降级后，问 AI："这套降级链有什么边界情况没考虑到？"AI 帮我指出来向量索引没加载时的空指针问题、ES 超时没 try-except 的问题。这种 review 比我自己盯着代码找效率高。

**还有个关键用法是让 AI 当面试官拷问我。** 就是现在这份 interview.md 的来源——我把项目代码给 AI，让它扮演面试官问深度问题，我回答，它点评。这比我自己闷头复习更能暴露盲点。比如 Q2 那个"LLM 既当裁判又当运动员"的问题，就是 AI 追问出来的，我自己复习不一定想得到这个角度。

**第四步：验证靠我自己跑，不靠 AI 说是对的。**

AI 写完代码说"测试通过"我从不信。我自己跑 `docker compose up`、上传一份真简历、走完整流程、看日志、查数据库。AI 会自信地编造不存在的 API 或参数，**验证环节必须人来做**。

比如之前遇到一个真实问题：服务端没跑最新代码，GitHub Actions 自动部署失败但没报错，服务器上 git 没拉到最新 commit。这种部署层的问题 AI 根本看不到，只能我自己 `docker compose logs` 查出来。

**总结一句话。**

我的 AI 开发流程是：**架构自己定、实现交给 AI、review 让 AI 互相拷问、验证自己跑**。核心原则是"决策权不外包"——AI 是加速器不是决策者，架构、验收、上线这三件事必须人做。

## Q6: WebSocket 断线重连——消息怎么不丢？怎么判断断线 vs 正常关闭？

**面试官原话**："断线期间消息不丢失"具体怎么实现的？客户端断线期间服务端 LLM 还在流式生成 token，这些 token 存哪？重连后全量重发还是增量补发？怎么判断"断线"还是"正常关闭"？客户端关浏览器，服务端那段还在跑的 LLM 怎么处理？

### 我的回答

**先说消息不丢的核心机制。**

我的做法是**写 Redis 在前、发 WebSocket 在后**。每条消息（问题、状态、完成、错误）都走 `_send_and_cache`：

```
1. push_message → 写入 Redis List (key = ws:msgs:{session_id})
2. send_json → 发 WebSocket
```

**顺序是关键**：先写 Redis 再发 WS。如果先发 WS 再写 Redis，发到一半连接断了，消息就丢了。先写 Redis，哪怕 WS 发送失败，消息还在 Redis 里。

**Redis List 而不是 Pub/Sub**——List 是持久化的，消息进队列就等着被消费；Pub/Sub 是 fire-and-forget，没人接就丢了。断线重连场景必须用持久化队列。

**重连后怎么补发？** 客户端重连时发 `{type: "start"}`，服务端调 `get_pending_messages`，用 Lua 脚本**原子地 LRANGE + DEL**——读出所有缓存消息并清空队列。原子操作防止并发重连导致重复消费。然后把这一批消息按顺序推给客户端。

**全量重发，不是增量。** 因为每条消息是完整的（一个完整的问题、一个完整的状态），不存在"半个 token"的问题。LLM 流式生成的 token 我在服务端聚合成完整消息再写 Redis，客户端拿到的永远是完整内容。

---

**再说"流式生成期间断线"的处理。**

这是个真问题——LLM 生成一个回答要几秒，这期间客户端断线了怎么办？

**我的处理是：服务端 LLM 任务继续跑完，结果写入 Redis。** 理由：

**第一，LLM 调用没法中断。** DeepSeek API 一旦开始流式返回，中途取消就浪费已生成的 token。让它跑完存 Redis，等客户端重连补发，成本最低。

**第二，面试场景下"那个问题"用户还是要看的。** 用户断线重连后，他需要看到刚才那个完整的问题才能回答。如果服务端中断了，重连后要重新生成，浪费一次 LLM 调用，而且可能生成不同的问题（temperature > 0）。

**代价是可能浪费一次生成**——如果用户彻底走了不重连，那次 LLM 调用的结果就浪费了。但 Redis TTL 1 小时自动清理，不会无限堆积，这个浪费可接受。

---

**最后说怎么判断"断线" vs "正常关闭"。**

**靠 WebSocket 协议本身。** FastAPI 的 `WebSocketDisconnect` 异常区分两种情况：

- **正常关闭**：客户端发 close frame（比如调用 `ws.close()`），服务端收到 code=1000。这种情况下 `_send_and_cache` 该发还是会发到 Redis，但知道客户端主动结束了。
- **异常断线**：TCP 连接断了（网络问题、浏览器崩溃），服务端收到的是非 1000 的 code 或者直接超时。

**实际处理上两者一视同仁**——都是停止当前连接的处理循环，缓存消息留在 Redis 等下次重连。区别在客户端：正常关闭客户端不会再重连，Redis TTL 1 小时后自动清理；异常断线客户端会触发自动重连（指数退避，最多 5 次），重连后补推缓存消息。

**为什么不做区分？** 因为服务端没法预判客户端会不会重连。统一按"可能重连"处理，缓存消息留着，让 Redis TTL 兜底清理。这比"猜客户端意图"简单且可靠。

**总结**：Redis List 持久化 + 先写后发保证不丢，Lua 原子读取保证不重，LLM 任务跑完存 Redis 保证重连能补，协议层 close code 区分语义但处理上一视同仁——四个设计点撑起消息可靠性。

---

### 可能的追问准备

- **"Redis 挂了怎么办？消息不就丢了？"** —— 会丢。Redis 是单点，挂了缓存消息没了。但面试消息同时写 DB（InterviewMessageORM），重连后能从 DB 恢复历史消息，只是断线期间的"未消费消息"丢了。极端情况可接受，Redis 挂是另一个问题（高可用、哨兵）。
- **"为什么用 Lua 脚本读 + 删？"** —— 防并发重连。如果分两步 LRANGE 然后 DEL，中间又来一个重连请求，会重复读到同一批消息。Lua 脚本在 Redis 单线程里原子执行，杜绝重复消费。
- **"多个客户端同时连同一个 session 怎么办？"** —— `active_connections` 是 `session_id -> set of websockets`，多连接都会收到消息。但面试状态在 DB 里只有一份，多客户端看到的是同一状态，没问题。
- **"LLM 流式生成期间，token 是不是边生成边发？"** —— 不是。我在服务端聚合完整回答再发，不在流式过程中往 WS 推 token。原因是断线重连补发完整消息比补发 token 流简单得多。代价是用户看不到"打字机效果"，但模拟面试不是聊天，等几秒看完整问题可接受。
- **"TTL 1 小时怎么定的？"** —— 经验值。面试一次 10-15 分钟，1 小时足够覆盖"断线→重试→重连"全过程。太短（比如 10 分钟）用户网络问题超过 10 分钟就丢了，太长（比如 24 小时）Redis 内存浪费。
- **"为什么不用消息队列（RabbitMQ/Kafka）？"** —— 杀鸡用牛刀。单个 session 的消息量就几条到几十条，Redis List 够用。消息队列是为高吞吐解耦设计的，我这是低吞吐、强一致、短生命周期的场景。

## Q7: 如果重做一次，最想改的架构决策是什么？

**面试官原话**：没有标准答案，想听对整套系统的反思。

### 我的回答

这个问题我想过很多次，**最想改的是把"评估器"和"决策器"分开——现在 evaluate_answer 节点既当裁判又当指挥，这是个架构层面的耦合，不是改个参数能修的。**

**现在的设计是这样的：** evaluate 节点调一次 LLM，LLM 返回 score + feedback + credible，然后节点内部用 if-else 判定 decision（follow_up / next_point / report）。评估和决策混在一个节点里。

**问题在哪？**

**第一，评估逻辑和业务规则耦合死了。** 我想调决策阈值——比如把 75 分改成 80 分，或者改成"连续两轮低于 60 分才切"——得改 evaluate.py 的代码。但评估逻辑（调 LLM、解析 JSON、Tool Calling）和决策逻辑（阈值判定、状态转移）是两回事，不应该绑在一起。

**第二，没法独立测试评估质量。** 我前面承认 LLM 评分不稳定，但我想做一致性测试时发现——评估和决策是一体的，我没法单独跑"100 条回答的评估"看分数分布，必须连着状态机一起跑。这让我对评估质量的改进无从下手。

**第三，决策规则没法热更新。** 阈值写死在代码里，想 A/B 测试不同阈值要改代码重启服务。生产环境这不可接受。

**如果重做，我会这样拆：**

```
evaluate_answer (纯评估)
    ↓ 输出 {score, feedback, credible, highlights, weaknesses}
decision_node (纯决策)
    ↓ 输入评估结果 + 状态，输出 decision
    ↓ 规则可配置，不写死在代码里
```

evaluate 节点只负责"这个回答怎么样"，decision 节点只负责"下一步干啥"。两件事解耦后：

- 评估器可以单独跑离线测试，拿一批回答跑 10 次看方差
- 决策规则可以做成配置，阈值、轮次上限、熔断条件都能热更新
- 想换评估策略（比如加个二次 LLM 校验、或者用规则评分替代）只改 evaluate 节点，不动 decision
- 想换决策策略（比如从"分数阈值"改成"贝叶斯可信度更新"）只改 decision 节点，不动 evaluate

**第二个想改的，是状态持久化。**

现在的状态恢复是 ws.py 和 mcp/server.py 各自实现的——WebSocket 断线靠 Redis + DB 恢复，MCP 靠内存 state。两条路径的恢复逻辑不一致，MCP 模式下服务重启状态就丢了。如果重做，我会把状态持久化下沉到 LangGraph 的 checkpoint 机制，让框架管这件事，而不是上层每个入口自己实现。这样任何接入方式（WebSocket、MCP、甚至未来的 HTTP）都自动获得状态恢复能力。

**第三个想改的，是知识库的版本化。**

现在知识库是 knowledge/ 目录下的 JSON 文件，改了就改了，没有版本概念。生产环境改知识库应该有灰度发布——先让 10% 的面试用新知识库，看评估质量有没有变化，再全量。现在的架构做不到。

**总结**：这三个反思的共同主题是**"关注点分离"**——评估和决策分离、状态管理和业务逻辑分离、知识库内容和运行时分离。做第一版时为了快把东西都揉在一起，跑起来之后才发现这些耦合在阻碍迭代。这是我对"如果重做"最真实的回答：不是技术选型错了，是边界没划清楚。

### 可能的追问准备

- **"为什么不一开始就这么拆？"** —— 第一版优先跑通，过度设计是坑。跑起来之后才知道哪些耦合真的痛——evaluate 决策耦合是做了一致性测试才发现的，状态持久化是 MCP 模式下重启丢状态才发现的。先做再改，比一开始猜需求靠谱。
- **"checkpoint 机制为什么没用？"** —— 当时为了赶进度用了自己实现的 state dict + DB 持久化，能用但不够通用。LangGraph 的 checkpoint 支持 PostgreSQL/Redis backend，自动管理版本，是更标准的方案。我承认这是技术债。
- **"评估和决策分离后，决策节点用规则还是 LLM？"** —— 规则。决策是状态转移，必须是确定性的，不能用 LLM。LLM 只负责评估（主观判断），规则负责决策（客观逻辑）。这正是分离的价值——把确定性还给决策层。
- **"知识库灰度发布怎么做？"** —— 知识库加版本号，DB 存多个版本，session 创建时绑定版本号，灰度比例通过配置控制。这是 standard 的 feature flag 思路。
- **"你觉得这些反思说明项目失败了？"** —— 不，能跑起来、能解决用户问题就是成功。但"能用"和"好维护"是两回事，反思是为了下一个项目做得更好，不是否定这个项目。
