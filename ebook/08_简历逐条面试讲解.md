## 项目简介

**简历写法：**
> AI 驱动的简历诊断与模拟面试系统：上传简历 → AI 深度诊断 → 多轮模拟面试 → 量化评估报告

**怎么讲：**
"这是一个端到端的 AI 面试系统。用户上传简历后，系统自动解析结构化数据，识别简历中的存疑点（比如数据夸大、技术深度不足），然后进行多轮模拟面试验证真实性，最后生成量化评估报告。"

**面试官问：** 和传统简历工具有什么区别？

**答：** 传统工具只做格式检查或关键词匹配。这个系统的核心是**Agent 驱动的多轮对话**——不是固定题库，而是根据候选人的回答动态追问，像真人在面试一样。

## LangGraph 状态机

**简历写法：**
> 使用 LangGraph 状态机编排面试流程，根据回答质量动态决策追问/切换/生成报告，三层防死循环机制

### 怎么讲

"面试流程用 LangGraph 状态机编排，核心是一个三节点循环：question（生成问题）→ collect（收集回答）→ evaluate（评估决策）。evaluate 节点根据评分返回三种决策：追问（follow_up）、切换存疑点（next_point）、生成报告（report）。"

"防死循环有三层保障：单点最多追问 3 轮就强制切换，用户可以随时跳过，LLM 异常时自动降级到规则评分保证流程不中断。"

### 面试官会问

**Q: 为什么用 LangGraph 而不是自己写循环？**

A: 自己写 `while True` + `if/else` 也能实现，但 LangGraph 的优势是：
1. 图结构天然可画流程图，方便调试
2. State 自动管理，加新字段不用改传递逻辑
3. 加新节点只加函数+连边，不影响已有代码

面试时可以补充："如果只是一个简单的 for 循环问 5 个问题，自己写确实够了。但这个系统有多种分支、工具调用、异常处理，用状态机更清晰。"

**Q: 状态存在哪里？断电了怎么办？**

A: 两个地方：
- 内存：当前请求的 State dict，请求结束就没了
- 数据库：`interview_states` 表存 checkpoint（当前存疑点、轮次、进度），`interview_messages` 表存完整对话历史

断电恢复：从数据库加载 checkpoint → 重建 State → 从断点继续面试。

**Q: 评估节点的评分标准是什么？**

A: LLM 从 5 个维度评分（0-100）：
1. 真实性 — 回答是否可信
2. 深度 — 技术理解程度
3. 完整性 — 是否回答了核心要点
4. 逻辑性 — 表达是否清晰
5. 技术准确性 — 和知识库交叉验证

决策逻辑（见 [evaluate.py](file:///d:/Code/ResumeAgent/backend/app/agent/nodes/evaluate.py)）：

```python
if current_round >= MAX_FOLLOW_UP or score >= 75:
    # 追问轮次到上限(3) 或 分数够好(≥75) → 当前点 resolved，切换下一个
    if next_index >= len(doubt_points):
        decision = "report"          # 所有点问完 → 生成报告
    else:
        decision = "next_point"      # 切换下一个存疑点
else:
    decision = "follow_up"           # 分数不够，继续追问，轮次 +1
```

另外如果 LLM 判定 `credible=false`，分数会被压到 ≤45；如果回答太短（<50 字符），分数会被压到 ≤40。

## RAG 三级检索 + Tool Calling

**简历写法：**
> RAG 三级混合检索（Embedding 向量 + Elasticsearch 全文 + 关键词匹配）与 Tool Calling 机制，自主调用知识库检索、简历字段查询、代码片段验证三个工具，根据上下文动态决定而非固定流程预加载

### 怎么讲（分两部分）

**RAG 部分：**
"知识库有 32 条高频面试题，覆盖 MySQL、Redis、Docker、Python、系统设计 5 个技术栈。检索有三层：第一层 Embedding 向量检索做语义匹配（比如'查询太慢'能匹配到'慢查询优化'），第二层 Elasticsearch 全文检索做关键词精确匹配，第三层简单关键词匹配兜底。三层串行降级，命中就停。"

**Tool Calling 部分：**
"面试 Agent 不是每次都预加载知识库，而是通过 Tool Calling 自主决定要不要调工具。系统定义了 3 个工具：知识库检索、简历字段查询、代码片段验证。LLM 根据对话上下文判断——比如候选人提到 B+树索引，LLM 决定调用知识库检索获取参考内容，然后生成更有针对性的追问。"

### 面试官会问

**Q: Embedding 和全文检索的本质区别？**

A: Embedding 把文字变成 1536 维向量，通过向量距离衡量**语义相似度**。"查询太慢"和"慢查询优化"词完全不同，但语义相近，向量距离近。全文检索基于**词频统计**（TF-IDF），看的是关键词是否出现。简单说：Embedding 理解**意思**，全文检索匹配**字面**。

**Q: 为什么不用向量数据库（ChromaDB/Milvus）？**

A: 数据量决定方案。项目只有 32 条面试题，用内存列表 + 余弦相似度就够了，零依赖零延迟。数据量到几万条以上才需要向量数据库。架构上已经解耦，换 ChromaDB 只需要改 `rag.py` 的检索函数。

**Q: Tool Calling 和直接在 prompt 里塞 RAG 结果有什么区别？**

A: 两个关键差异：
1. **谁决定检索**：固定 RAG 是代码硬编码每次都检索，Tool Calling 是 LLM 自主决定需要时才调用
2. **调用范围**：固定 RAG 只能在生成问题前检索，Tool Calling 在提问和评估阶段都能调用（比如评估时调用代码验证工具）

**Q: DeepSeek 的 Tool Calling 和 OpenAI 有什么不同？**

A: OpenAI 返回结构化的 `tool_calls` 字段，DeepSeek 有时把工具调用输出为文本内容（DSML 格式）。我们做了两层解析：优先取结构化 `tool_calls`，没有的话用正则从文本中提取工具名和参数。这是一个兼容层，让同一套代码能跑多个 LLM。

## MCP 技能包

**简历写法：**
> 利用 MCP 协议将系统封装为标准化 Agent Skills，定义 7 个工具接口，支持主流 AI 编程工具直接调用

### 怎么讲

"把整个系统封装成 MCP Server，定义了 7 个工具：诊断简历、开始面试、提交回答、跳过问题、结束面试、获取报告、列出会话。Claude Code、Cursor 这些 AI IDE 通过 stdio 协议直接调用，不需要启动后端服务。MCP Server 直接调用内部模块，是独立进程。"

### 面试官会问

**Q: MCP 和 REST API 的区别？**

A: MCP 是**给 AI 用的协议**，REST API 是**给人用的**。REST 需要程序员读文档、写代码调用。MCP 的 AI 客户端通过 `list_tools()` 自动发现工具，通过 `description` 理解用途，通过 `inputSchema` 知道参数格式。相当于给 AI 的"即插即用"接口。

**Q: 为什么不直接用 REST API？**

A: REST API 也能用，但 MCP 的优势是**标准化**。定义一次工具，所有 MCP 客户端（Claude Code、Cursor、Windsurf、Continue 等 10+ 工具）都能直接调用。不用为每个客户端写适配代码。

**Q: MCP Server 怎么做到不依赖 FastAPI？**

A: MCP Server 直接 import 内部模块调用，不走 HTTP。`parse_task.apply()` 直接执行解析，`generate_question()` 直接生成问题，数据库直连 PostgreSQL。FastAPI 是给前端用的 MCP 是给 AI 用的，两条路径独立。

## WebSocket 实时通信

**简历写法：**
> 基于 WebSocket 实现实时面试对话，支持断线自动重连与 Redis 缓存，LLM 响应流式推送，断线期间消息不丢失

### 怎么讲（分三层）

**WebSocket 实时通信：**
"面试用 WebSocket 双向通信，不是 HTTP 轮询。用户发答案、服务端推问题，都在一个长连接里完成。"

**流式推送：**
"LLM 生成问题时用流式模式，每生成一个 token 就通过 WebSocket 推给前端，用户看到文字逐渐出现。实现上用 asyncio.Queue 跨线程传递——后台线程跑 LLM 流式生成器，主事件循环从队列取 chunk 推给客户端。"

**断线恢复：**
"断线恢复有三层保障：
1. 前端消息队列 — 用户发答案时 WS 断开，消息存 JS 数组，重连后自动发送
2. Redis 消息缓存 — 后端生成了问题但客户端断线，消息存 Redis List，重连时补推
3. 数据库持久化 — 页面刷新或长时间断线，从 interview_messages 表恢复完整历史"

### 面试官会问

**Q: 为什么不用 SSE？**

A: WebSocket 和 SSE 都能实现流式推送，但 WebSocket 是双向的——发答案和收问题用同一个连接。SSE 是单向的（服务端→客户端），发答案需要额外的 HTTP 请求。面试场景需要双向通信，WebSocket 更合适。

**Q: 流式推送的延迟瓶颈在哪？**

A: 三个环节：
1. LLM 首 token 延迟（200-500ms，取决于 API）
2. 网络传输（WebSocket 逐 chunk 推送，几乎无额外延迟）
3. 前端渲染（React 状态更新，约 16ms/帧）

瓶颈在 LLM API。流式推送让用户**感知延迟**从 3 秒降到 200ms。

**Q: 断线重连为什么用指数退避？**

A: 防止重连风暴。如果服务端故障，100 个客户端同时重连，固定间隔 1 秒会每秒 100 次请求。指数退避（1s→2s→4s→8s→16s）分散请求，给服务端恢复时间。最多重试 5 次，超过提示用户刷新。

**Q: Redis 缓存的消息格式是什么？**

A: Redis List，key 是 `ws:msgs:{session_id}`，value 是 JSON 序列化的消息数组。用 Lua 脚本原子执行 LRANGE + DEL，保证读取和清除是原子操作，防止重复消费。TTL 1 小时自动过期。
