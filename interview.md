## Q1: LangGraph 状态机的节点和边怎么设计？"回答质量"怎么量化？"三层防死循环"是哪三层？

**面试官原话**：节点和边是怎么设计的？条件边的路由逻辑是什么？"质量"怎么量化判定？三层防死循环具体是哪三层？LLM 在节点间来回跳转死循环怎么防？

### 我的回答

**先说节点和边。**

我的状态机有 4 个节点：`generate_question`（生成问题）、`collect_answer`（收集回答）、`evaluate_answer`（评估回答）、`generate_report`（生成报告）。

核心循环是：生成问题 → 收集回答 → 评估 → 条件分支。条件边只有一条，在 evaluate 之后，路由函数只返回两个目标——`generate_question`（继续问）或 `generate_report`（结束）。

这里有个设计点我想强调：**graph 层只管"继续 or 结束"，不管"是追问还是切换存疑点"**。follow_up 和 next_point 的区分在 evaluate 节点内部决定，通过 `decision` 字段透传给上层 WebSocket handler。这样图保持简单，加节点不影响已有边。

**再说质量怎么量化。**

用 LLM 从 5 个维度评估：真实性、深度、完整性、逻辑性、技术准确性，输出 0-100 的综合分。评估节点支持 Tool Calling——LLM 会主动调知识库检索工具验证候选人说的技术对不对。

决策阈值是 **75 分**：分数 ≥75 或追问到 3 轮上限，当前存疑点判定 resolved，切下一个点；否则继续追问。

但光看分数不够，加了两个硬性压制：LLM 判定回答不可信（`credible=false`）→ 分数压到 ≤45；回答太短（<50 字符）→ 分数压到 ≤40。确保"洋洋洒洒但没料"和"一句话敷衍"都蒙混不过去。

**最后说三层防死循环，这是 Agent 最容易翻车的地方。**

**第一层，节点级——单点追问上限。** 同一个存疑点最多追问 3 轮，到 3 轮强制切下一个点。防的是"LLM 死活不满意候选人回答，无限追问"。

**第二层，图级——会话硬熔断。** 路由函数检查四个全局上限：连续 LLM 异常 3 次、单场最多 20 个问题、Token 上限 5 万、单次 LLM 调用 15 秒超时。只要触上限就强制结束，不管 evaluate 返回什么。防的是"节点决策层失灵"。

**第三层，LLM 降级——避免 LLM 不确定性。** LLM 不可用、超时、JSON 解析失败时，自动降级到规则评分，按回答长度给分。规则逻辑确定，永远不会让状态机卡住。

**总结一句话**：节点级防单点死循环、图级防会话级失控、降级层防 LLM 不确定性——三层从局部到全局、从概率到确定，把 LLM 这种不可控的东西框在一个可预测的状态机里。这是我对 Agent 工程化最核心的理解。

---

### 可能的追问准备

- **"为什么阈值是 75 分？"** —— 经验值，实际跑下来 75 分能区分"有细节支撑"和"泛泛而谈"，可调。
- **"规则评分会不会误判？"** —— 会，长度不代表质量，但作为 LLM 挂掉时的兜底够用，总比流程卡死强。
- **"为什么 MAX_FOLLOW_UP 是 3？"** —— 面试场景下 3 轮足够判断候选人懂不懂，再问下去浪费双方时间。
- **"为什么不把 follow_up/next_point 做成条件边的两个目标？"** —— 图结构塞业务逻辑会变复杂，加节点要改边；放节点里图保持简单，业务逻辑也好测。

## Q2: LLM 评分的稳定性和可信度问题

**面试官追问**：你的评估器本身也是 LLM 调用。同一个回答跑两次分数可能差 10 分，75 分阈值在波动面前很脆弱。`credible=false` 本身也是 LLM 判的，用不稳定的判定修正不稳定的判定，是不是循环依赖？为什么没做 majority vote？

### 我的回答

您这个问题问到点子上了，我先承认——这是整个系统最薄弱的一环，LLM 评分不稳定是 Agent 系统的本质矛盾，我没有彻底解决，但做了四件事把它限制在可接受范围。

**第一，temperature 压到 0.3。** 评估节点是整个系统里温度最低的，生成问题用 0.7、闲聊用 0.9，评估刻意压低随机性。这是最直接的降波动手段。

**第二，引入客观锚点。** 评估节点支持 Tool Calling，LLM 会调知识库验证候选人说的技术对不对。比如候选人说"B+ 树适合范围查询"，LLM 调知识库查到这条是对的，这就不是主观判断了，是客观事实。工具结果回喂给 LLM，相当于外部约束，比纯主观评分稳。

**第三，用布尔判定兜连续值。** 我承认 `credible` 也是 LLM 判的，但它和 score 不是一回事——score 是 0-100 容易飘，credible 是布尔值，只有"明显不可信"才标 false，相当于否决权而非评分权。用稳的去约束飘的，逻辑成立。而且我还有 `answer_too_short` 这个 100% 规则化的判定，完全不依赖 LLM。

**第四，三层防死循环本身就是对评估不稳定的兜底。** 即使某次评分抽风切错了点，最坏后果是这个存疑点没问透，不影响整体面试完成。我接受这个代价——模拟面试不是司法判决，偶尔飘一次可接受。

**为什么没做 majority vote？** 成本和延迟。单场面试 10-15 分钟，每个回答跑 3-5 次评估，延迟翻倍 Token 翻倍，体验差。

**我承认的局限：** 如果做生产级，我会做两件事——一是离线跑评估一致性测试，拿 100 条回答跑 10 次看方差，方差大的加规则约束；二是关键决策（最终报告）用更强模型做 judge，单次评估用轻量模型控成本。

**总结**：LLM 评分不稳定我没法消灭，但用低温、客观锚点、布尔压制、三层兜底，把它限制在"偶尔切错一个点"的范围内，不让它击穿整个流程。

---

### 可能的追问准备

- **"score 阈值 75 是怎么定的？"** —— 经验值，跑下来 75 分能区分"有细节支撑"和"泛泛而谈"。其实 75 分不是死线，3 轮上限也是切点，两个条件是 OR 关系。
- **"Tool Calling 自己失败了怎么办？"** —— 工具执行 try-catch，失败返回错误字符串，LLM 看到错误会换方式回答，不影响主流程。
- **"为什么 temperature 0.3 不是 0？"** —— 0 太死板，评估需要一点灵活性判断"有创意的回答"；0.3 是 trade-off。

## Q3: Redis + Lua 滑动窗口限流，为什么用 Lua？为什么选滑动窗口？

**面试官原话**：滑动窗口限流的核心是"读取计数 + 判断 + 写入"几步，分开执行会出什么问题？Lua 解决了什么？滑动窗口和固定窗口区别？OJ 系统限流对象是什么？为什么选滑动窗口不选固定窗口或令牌桶？

### 我的回答

**先说为什么用 Lua。**

滑动窗口限流的核心逻辑是三步：读当前窗口计数 → 判断是否超限 → 写入新计数。如果这三步用三条 Redis 命令分开执行，**一定会出并发问题**。

举个具体场景：限流 100 次/分钟。两个请求几乎同时到达，都读到 count=99，都判断"没超限"，都写 count+1，结果 count 变成 100，但实际应该是 101 第二个请求要被拒。**这是典型的 check-then-act 竞态条件。**

Lua 脚本解决两件事：

**第一，原子性。** Redis 是单线程执行命令的，一个 Lua 脚本在执行期间不会被其他命令打断。把三步塞进一个脚本，相当于一个不可分割的操作，竞态条件从根上消除。

**第二，减少网络往返。** 三条命令是三次 RTT，Lua 脚本一次 RTT 搞定。高并发下这个差距很明显。

我还在脚本里用 ZSET 实现滑动窗口——key 是用户 ID + 接口，score 是时间戳，value 是唯一请求 ID。每次先 `ZREMRANGEBYSCORE` 清掉窗口外的旧数据，再 `ZCARD` 数窗口内数量，判断是否放行，放行就 `ZADD` 加一条。整个逻辑在脚本里一气呵成。

**再说为什么选滑动窗口。**

三种主流限流算法对比：

**固定窗口**：按自然时间分桶（比如每分钟一个桶）。致命问题是**窗口边界突发**——0:59 来 100 个请求，1:01 又来 100 个请求，两秒内 200 个请求，但每个窗口看都没超限。OJ 系统不能接受这个，一个用户在边界刷 200 次提交会把判题队列打爆。

**令牌桶**：允许突发流量，桶满了瞬间放行。适合**流量整形**场景（比如 API 网关放宽短时突发）。但 OJ 不需要放宽突发——提交就是提交，突发对系统没好处。

**滑动窗口**：没有边界突发问题，任意时刻往回看 60 秒，计数都是准确的。对 OJ 这种"严格限制单位时间操作次数"的场景最合适。

**限流对象**：我的 OJ 是**按用户 + 按接口**双维度限。提交代码接口 10 次/分钟（防恶意提交打爆判题机），登录接口 5 次/分钟（防暴力撞密码），通用查询接口 60 次/分钟。用 AOP + 自定义注解，方法上加 `@RateLimit(limit=10, window=60)` 就生效，业务代码零侵入。

**总结**：Lua 解决并发安全（原子性 + 省 RTT），滑动窗口解决边界突发，AOP 解决接入成本——三个问题各司其职。

### 可能的追问准备

- **"Lua 脚本挂了 Redis 会不会阻塞？"** —— 会的，Redis 单线程，Lua 脚本执行期间其他命令排队。所以脚本要短小，我那个脚本就几行，执行时间微秒级。千万别在脚本里做重逻辑。
- **"集群模式下 Lua 有什么坑？"** —— Redis Cluster 下 Lua 脚本里所有 key 必须在同一 slot，不然报错。我的做法是 key 里带 `{user_id}`  hash tag，强制路由到同一节点。
- **"为什么不用 Redis 的 INCR + EXPIRE 做固定窗口？"** —— 能用，但有边界突发问题，OJ 场景不能接受。INCR 方案适合对突发不敏感的场景（比如统计 PV）。
- **"限流被触发后怎么处理？"** —— 抛自定义异常，全局异常处理器转成 429 状态码 + Retry-After 头，前端收到 429 提示"操作过于频繁"。
- **"为什么 AOP 而不是拦截器？"** —— 拦截器只能拦 Controller 层，AOP 可以拦任意层（Service、方法），且注解更直观，`@RateLimit` 贴在方法上语义清晰。

## Q4: RAG 三级检索的结果怎么融合？向量和全文各自擅长什么？

**面试官原话**：三级检索的融合策略是什么——RRF、加权分数、还是去重拼接？为什么选这个？向量检索和全文检索各自擅长什么？"消息队列怎么保证不丢"和"Kafka 的 ISR 机制"分别哪一路能命中？

### 我的回答

**融合策略用 RRF（Reciprocal Rank Fusion），但只融合向量+ES 两路，关键词作为兜底。**

具体逻辑：向量检索和 ES 全文**并行召回**，各自返回 top-2K，然后用 RRF 公式 `score = Σ 1/(k + rank)` 融合排序，取 top-K。k 取 60，论文经验值。

**为什么只用 RRF 不用加权分数？** 三路检索器的分数量纲不一致——向量是余弦相似度 0~1，ES 是 BM25 分数可能 0~20。加权融合要先归一化，归一化方法又是调参坑。RRF 只看排名不看绝对分数，量纲无关，省了归一化。

**为什么关键词不参与融合？** 关键词是子串匹配，几乎所有结果都能命中，"排名"权重虚高会污染融合分数。所以只让向量+ES 融合，关键词在前两路都挂了时才兜底。

**RRF 的核心价值**：同一条结果被两路都命中，分数累加排名更靠前——多路共识的结果更可靠。这是融合比单路召回强的本质。

**降级链也保留了**：我做了 `RETRIEVAL_MODE` 配置，支持 `fusion`（默认，双路 RRF）/ `cascade`（串行降级老逻辑）/ `vector_only` 三种模式，便于 A/B 测试和回滚。

---

**第二个问题，向量和全文各自擅长什么。**

**向量检索擅长语义相关但字面无关的查询。** 比如候选人简历写"我用过 Kafka"，面试官问"消息队列怎么保证消息不丢"——"消息队列"和"Kafka"、"不丢"和"ack 机制"字面差很远，全文检索抓不住，但向量把它们映射到相近的语义空间，能命中。本质是 embedding 把词映射到向量空间后，同义词、上下位词距离近。

**全文检索擅长术语精确匹配。** 比如"Kafka 的 ISR 机制"、"MVCC"、"B+ 树"——这些是专有名词，就是字面那几个字，向量检索反而可能因为语义相近漂到别的地方（比如 ISR 漂到"消息可靠性"的泛泛内容），全文检索字面精确命中更准。

**关键词匹配是兜底，前两个都挂了才用。** 零依赖、零延迟、永远不会挂。LLM 和 ES 都不可用时保证流程能走完。

**总结**：向量管语义、ES 管字面，双路 RRF 融合取共识；关键词管兜底，前两路都挂了才上。三种检索器各司其职，融合策略让多路共识的结果浮上来。

### 可能的追问准备

- **"为什么不用向量数据库？"** —— 32 条数据用内存列表 + 余弦相似度，零依赖零延迟。上向量数据库（ChromaDB、Milvus）要部署、要维护、要索引，收益不抵成本。数据量上千再换。
- **"RRF 的 k 为什么是 60？"** —— 论文经验值。k 越大排名靠后的结果分数差距越小（被压缩）；k 越小头部结果优势越明显。60 是平衡点，既不让第 1 名压倒一切，也不让排名 10 和排名 1 没区别。数据量小没做实测调优。
- **"embedding 模型为什么用 deepseek-embedding？"** —— 复用 LLM 厂商的 embedding 接口，不用额外部署模型。中文场景效果够用，且和 DeepSeek LLM 配套。
- **"向量检索的 top_k 为什么是 3？"** —— 注入 LLM prompt 的上下文不能太长，3 条已经够 LLM 参考。top_k 太大会稀释相关性、增加 token 成本。注意融合时每路召回 top_k * 2，给融合留余量。
- **"如果向量检索返回的是垃圾结果怎么办？"** —— RRF 融合能通过多路投票消除单路噪声——向量返回垃圾但 ES 命中好结果，融合后 ES 的好结果排名更靠前。这是融合比降级强的关键。
- **"为什么保留 cascade 模式？"** —— A/B 测试和回滚。融合是新版默认，但老逻辑不能丢，万一融合效果不如降级（虽然理论上不应该），一行配置切回去。生产环境必须有回滚手段。
