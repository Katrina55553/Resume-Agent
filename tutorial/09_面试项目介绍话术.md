# 面试项目介绍话术

本文件聚焦**"面试时怎么开口介绍这个项目"**——按场景给话术模板，对应到代码细节方便追问。

> 关于每个技术亮点的深度问答，见 `01~06` 与 `07_综合面试题汇总.md`。
> 关于简历每条怎么讲，见 `08_简历逐条面试讲解.md`。

## 一、开场禁忌与建议（先避坑再开口）

### ❌ 不要这样讲

1. **背简历式**："项目用了 React、FastAPI、LangGraph、PostgreSQL、Redis、Docker……"（流水账，没信息量）
2. **过度自夸式**："这是一个非常先进的、行业领先的 AI 系统……"（空洞）
3. **纯技术堆砌**："我用了 asyncio.Queue、call_soon_threadsafe、DSML 正则……"（一上来就陷细节）
4. **模糊业务**："就是一个 AI 面试工具。"（没说清价值）

### ✅ 推荐结构（5 步开口法）

1. **一句话业务价值**（解决什么问题）
2. **一句话技术栈**（用了什么）
3. **3-4 个技术亮点**（按优先级讲，每个一句话）
4. **一个最有挑战的点**（留钩子让面试官追问）
5. **"您想从哪个方向深入聊？"**（交回主动权）

## 二、三档话术模板（按时间选用）

### 1. 30 秒电梯版（HR 面 / 破冰）

> "我做了一个 AI 驱动的简历诊断和模拟面试系统。用户上传简历后，AI 自动解析结构化数据、识别简历中的存疑点，然后通过 WebSocket 多轮对话进行模拟面试，最后生成量化评估报告。技术上用 LangGraph 编排 Agent 状态机、RAG 三级检索 + Tool Calling 让 Agent 自主调工具、MCP 协议把系统封装成标准化技能包，全栈从 React 前端到 Docker 部署独立完成。"

**关键点：** 一句话说清业务 + 一句话点出技术亮点 + 强调全栈独立完成。

---

### 2. 2 分钟标准版（技术面开场，**推荐**）

> "这是一个端到端的 AI 面试系统，核心链路是：上传简历 → AI 解析结构化 → 诊断存疑点 → 多轮模拟面试 → 量化评估报告。
>
> 技术栈是 React + FastAPI + LangGraph + DeepSeek + PostgreSQL + Redis + Elasticsearch + Docker。
>
> 我在项目里主要做了四件事：
>
> **第一，用 LangGraph 状态机编排面试流程。** 设计了 question → collect → evaluate → 条件分支 → report 的状态图，evaluate 节点根据评分返回 follow_up / next_point / report 三种决策。为了防止死循环，做了三层保护：单点最多追问 3 轮、用户可主动跳过、LLM 异常自动降级到规则评分。
>
> **第二，RAG 三级检索 + Tool Calling。** 知识库 32 条高频面试题，串行降级检索：Embedding 向量 → Elasticsearch 全文 → 关键词匹配。同时定义了 3 个工具（知识库检索、简历字段查询、代码片段验证），让 Agent 自主决定调用，而不是固定流程预加载。
>
> **第三，WebSocket 实时通信 + 断线恢复。** LLM 流式生成用 asyncio.Queue 跨线程推 chunk，首字延迟从 3 秒降到 200ms。断线恢复三层保障：前端消息队列缓存未发送消息、Redis 缓存未消费消息、PostgreSQL 持久化完整对话历史。
>
> **第四，把系统封装成 MCP Server。** 定义 7 个工具，Claude Code、Cursor 这些 AI IDE 通过 stdio 协议直接调用，不需要启动后端服务。"
>
> 最后是部署：Docker Compose 编排 6 个服务（Nginx + FastAPI + Celery + PostgreSQL + Redis + Elasticsearch），全部带健康检查。

**讲完留一个钩子**："您想从哪个方向深入聊？"——把主动权交回去，避免一上来就被打断。

### 3. 5 分钟深度版（深度技术面 / 终面）

按 **业务 → 架构 → 难点 → 收获** 四段式：

#### 第 1 段：业务背景（30 秒）

> "传统简历工具只做格式检查或关键词匹配，无法识别'简历水分'。这个系统用 Agent 驱动的多轮对话，像真人面试官一样根据回答动态追问，验证简历真实性。"

#### 第 2 段：整体架构（90 秒）

> "前端 React + Zustand 管理状态，HTTP 走文件上传和诊断轮询，WebSocket 走实时面试。
>
> 后端 FastAPI 异步框架，三条关键路径：
> 1. **同步路径**：简历四重校验（扩展名 + MIME + 魔数 + 大小）→ Celery 异步解析 → AI 结构化提取或规则降级
> 2. **诊断路径**：DeepSeek 识别存疑点，按高/中/低优先级分类
> 3. **面试路径**：LangGraph 状态机驱动，WebSocket 实时通信，最终生成报告
>
> 数据层：PostgreSQL 存会话和面试状态、Redis 做 Celery Broker 和 WebSocket 消息缓存、Elasticsearch 做全文检索。"

#### 第 3 段：技术难点 + 解法（150 秒）

挑 2-3 个讲透，**每个难点必须按 STAR 法则：场景 → 问题 → 方案 → 结果**。

**难点 1：DeepSeek Tool Calling 格式不兼容 OpenAI**

> "场景：用 DeepSeek 做 Tool Calling，期望它返回结构化 tool_calls 字段。
> 问题：DeepSeek 有时把工具调用输出为文本（DSML 格式 `<｜｜DSML｜｜tool_calls>`），按 OpenAI 标准解析会漏掉。
> 方案：做了两层兼容解析——优先取结构化 tool_calls，没有的话用正则从文本提取工具名 + 附近 JSON 参数。还做了参数完整性校验，缺必填参数的工具调用直接丢弃，避免下游崩溃。
> 结果：同一套代码能跑 OpenAI 和 DeepSeek，切换模型只改配置。"

**难点 2：LLM 流式生成跨线程推送**

> "场景：WebSocket 面试要逐字推送，让用户感知到打字效果。
> 问题：DeepSeek SDK 的流式生成是同步迭代器（`for chunk in stream`），不能在 FastAPI 异步函数里直接 await。
> 方案：用 asyncio.Queue 做跨线程通道——后台线程跑同步生成器，每生成一个 chunk 用 `loop.call_soon_threadsafe` 推入队列，主事件循环 await queue.get 后通过 WebSocket 发送。
> 结果：首字延迟从等完整生成的 3 秒降到 200ms，用户体验显著提升。"

**难点 3：面试状态断线恢复**

> "场景：用户面试到一半网络断了或刷新页面。
> 问题：WebSocket 是无状态长连接，断线后服务端不知道客户端是否还在，已生成的问题会丢失。
> 方案：三层保障——前端消息队列缓存断线期间的发送、Redis List 缓存服务端未消费消息（用 Lua 脚本原子 LRANGE+DEL 防重复消费，TTL 1 小时）、PostgreSQL 持久化完整对话历史用于页面刷新恢复。重连用指数退避（1s→2s→4s→8s→16s）防重连风暴。
> 结果：断线 1 小时内都能恢复，不丢消息。"

#### 第 4 段：收获（30 秒）

> "这个项目让我系统掌握了 Agent 工程化的几个关键点：状态机编排 vs 直接写循环的取舍、Tool Calling 的兼容性陷阱、流式推送的跨线程模式、断线恢复的多层兜底思维。也让我对生产环境的稳定性有了更深理解——每层都要有降级方案，LLM 不可用时也要保证流程能走完。"

## 三、按面试官类型调整侧重

### HR 面 / 业务面

> "这个系统解决的核心问题是简历水分检测。传统工具只能做格式检查，我做的系统用 AI Agent 多轮对话模拟面试，根据候选人回答动态追问——比如候选人写了'优化了 MySQL 查询'，Agent 会追问'用什么手段优化？B+树还是覆盖索引？'，根据回答深度判断真实性。

**侧重实现细节：**

---

### 技术面（高级 / 架构师）

**侧重架构决策和取舍：**

讲完标准版后，主动抛出几个**决策点**让面试官追问：

1. **为什么状态机用 LangGraph 而不是自己写循环？**
   - 答：图结构可视化、State 自动管理、加节点不影响已有逻辑。但如果只是简单 for 循环，自己写也够——我这个项目分支多、有 Tool Calling 和异常处理，状态机更清晰。

2. **为什么 RAG 三级检索是串行降级而不是融合？**
   - 答：避免结果重复、逻辑简单易调试、性能好（命中就停）。数据量小（32 条）时这是最优解；数据量大时可以考虑 RRF（Reciprocal Rank Fusion）融合。

3. **为什么 WebSocket 用 asyncio.Queue 而不是直接异步迭代？**
   - 答：DeepSeek SDK 的流式是同步迭代器，不能直接 await。Queue 是同步到异步的标准跨线程通道。

4. **MCP 和 REST API 的边界在哪？**
   - 答：REST 给前端用，MCP 给 AI 用。两条路径独立，MCP Server 直接调内部模块不走 HTTP，减少一次网络跳转。

主动抛决策点比被动等问更显功底。

---

### 终面 / 总监面

**侧重全局观和成长性：**

> "这个项目最大的价值不是单个技术点，而是让我理解了**生产级 Agent 系统的工程化**。
>
> 第一是**降级思维**——每层都有兜底：LLM 挂了用规则、ES 挂了用关键词、WebSocket 断了用 Redis 缓存。生产环境不能假设依赖永远可用。
>
> 第二是**成本意识**——多模型路由，简单任务用轻量模型、复杂评估用旗舰模型，单次面试成本降 50%+。
>
> 第三是**可观测性**——LLM 调用记成本日志、WebSocket 发 chunk 数日志、面试决策记 decision 字段。出问题能定位到哪一步。"

## 六、收尾模板

### 如果面试官说"挺好的，没别的问题了"

> "这个项目还有很多可以优化的地方，比如多模态简历解析、面试回放、A/B 测试 prompt 等，如果您感兴趣我们可以继续聊。"

### 如果被问"你觉得这个项目还有什么不足"

> "主要三点：
> 1. **缺少用户系统**——目前没做 JWT 认证和多租户隔离，生产化需要补
> 2. **缺少监控告警**——LLM 调用失败率、Token 消耗、面试完成率这些指标没接 Prometheus
> 3. **知识库太小**——32 条面试题覆盖面有限，需要扩展到几百条并接入向量数据库"

主动说不足 = 有反思能力，比被问出来强。
